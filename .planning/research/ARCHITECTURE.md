### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| Pages | Route-based static pages (Home, Services, About, Contact, Blog index) | `.astro` files in `src/pages/` |
| Blog | Markdown/MDX content rendered as HTML at build time | `.md` files in `src/content/blog/`, Astro Content Collections |
| Layouts | Shared page shells (header, footer, nav, meta) | `BaseLayout.astro`, `BlogPostLayout.astro` |
| UI Components | Reusable visual elements (buttons, cards, CTAs, hero sections) | `.astro` components, zero JS by default |
| Build Layer | Compiles everything to static HTML/CSS/JS, optimizes images | Astro SSG mode + Vite bundler |
| Calendly Embed | Booking widget loaded client-side via Calendly's JS snippet | Inline embed on Contact page, pop-up widget as persistent CTA |
| Form Service | Receives contact form submissions, sends email notifications | Third-party API (Formspree, Web3Forms, or Netlify Forms) |
| Analytics | Page view and conversion tracking | Optional: Plausible, Fathom, or Google Analytics script |
| Static Host | Serves pre-built files from CDN edge nodes | Netlify, Vercel, or Cloudflare Pages |

## Recommended Project Structure

```
src/
├── assets/               # Images, brand assets (processed by Vite)
│   ├── images/           # Site images, team photos
│   └── brand/            # Logo variants, favicons
├── components/           # Reusable UI components
│   ├── layout/           # Header.astro, Footer.astro, Nav.astro
│   ├── sections/         # Hero.astro, ServiceCard.astro, CTA.astro
│   ├── blog/             # PostCard.astro, TagList.astro
│   └── forms/            # ContactForm.astro, CalendlyEmbed.astro
├── content/              # Content Collections (Astro built-in)
│   ├── blog/             # Blog posts as .md/.mdx files
│   └── config.ts         # Content collection schemas
├── layouts/              # Page shells
│   ├── BaseLayout.astro  # Global head, header, footer
│   └── BlogPost.astro    # Blog post wrapper with meta
├── pages/                # File-based routing
│   ├── index.astro       # Home page
│   ├── services.astro    # Services page
│   ├── about.astro       # About/Team page
│   ├── contact.astro     # Contact form + Calendly embed
│   └── blog/
│       ├── index.astro   # Blog listing page
│       └── [...slug].astro  # Dynamic blog post routes
├── styles/               # Global styles
│   └── global.css        # Base styles, CSS variables, typography
└── data/                 # Static data (services list, team info)
    └── services.ts       # Structured service offering data
public/
├── robots.txt
├── sitemap.xml           # (or generated by @astrojs/sitemap)
└── fonts/                # Self-hosted font files if needed
```

### Structure Rationale

- **components/:** Organized by function (layout, sections, blog, forms) rather than by page. Most consulting site components are reused across pages (CTAs appear on every page, service cards on home and services).
- **content/:** Astro Content Collections provide type-safe frontmatter validation, built-in Markdown rendering, and automatic slug generation. Blog posts live here as plain Markdown files.
- **layouts/:** Separated from components because layouts define the page shell, not reusable blocks. Two layouts covers all pages: base for static pages, blog post for articles.
- **data/:** Structured data files (services, team members) kept separate from components. This makes content updates easy without touching component logic.
- **public/:** Unprocessed static assets served as-is (fonts, robots.txt). Brand images go in `src/assets/` instead so Astro can optimize them.

## Architectural Patterns

### Pattern 1: Zero-JS-by-Default Pages

**What:** Astro renders all components to static HTML at build time. No JavaScript is shipped unless explicitly opted in with `client:*` directives.
**When to use:** Every page on this site. The only interactive elements needing JS are the Calendly embed and the contact form submission.
**Trade-offs:** Extremely fast page loads and perfect Lighthouse scores. The trade-off is that any interactivity requires deliberate opt-in, which is exactly right for a marketing site where 95% of content is static.

### Pattern 2: Astro Islands for Third-Party Widgets

**What:** Interactive components (Calendly, form with client-side validation) are isolated "islands" that hydrate independently. The rest of the page remains static HTML.
**When to use:** The Calendly embed and contact form are the two islands on this site.
**Trade-offs:** Each island loads its own JS independently, so there is no framework overhead for the rest of the page. The downside is that islands cannot easily share state with each other, but that is irrelevant for this site since the form and booking widget are independent.

**Example:**
```astro
---
// ContactForm island - only this component ships JS
import ContactForm from '../components/forms/ContactForm.astro';
import CalendlyEmbed from '../components/forms/CalendlyEmbed.astro';
---
<section class="contact-section">
  <!-- Static content: zero JS -->
  <h2>Get in Touch</h2>
  <p>Ready to explore how AI can transform your business?</p>

  <!-- Island 1: Form with client-side validation -->
  <ContactForm />

  <!-- Island 2: Calendly inline embed -->
  <CalendlyEmbed url="https://calendly.com/oriflect/discovery" />
</section>
```

### Pattern 3: Content Collections for Blog

**What:** Astro Content Collections define a schema for blog posts (title, date, tags, description) and validate frontmatter at build time. Posts are plain Markdown files.
**When to use:** All blog content. This gives type safety, automatic RSS feed generation, and easy querying (sort by date, filter by tag).
**Trade-offs:** Content must be committed to the repo (no CMS). This is fine given the project's explicit "no CMS" decision. If a CMS is ever needed later, Astro supports headless CMS integrations without restructuring.

## Data Flow

### Page Render Flow (Build Time)

```
Markdown/Astro Source Files
    |
    v
Astro Build (SSG)
    |
    +-- Reads content collections (blog posts)
    +-- Resolves layouts and components
    +-- Optimizes images (astro:assets)
    +-- Generates HTML for each route
    |
    v
Static Output (dist/)
    |
    v
CDN / Static Host --> Browser
```

### Contact Form Submission Flow (Runtime)

```
User fills form on /contact
    |
    v
Client-side validation (HTML5 + optional JS)
    |
    v
Form POST to third-party endpoint
(e.g., https://formspree.io/f/{form-id})
    |
    +-- Formspree validates, stores, sends email notification
    |
    v
Redirect to /contact?success=true or show inline confirmation
```

### Calendly Booking Flow (Runtime)

```
User clicks "Book a Discovery Call" CTA
    |
    v
Calendly JS widget loads (inline embed or pop-up)
    |
    v
User selects time slot within Calendly UI
    |
    v
Calendly handles all booking logic, confirmation emails
(No data touches the Oriflect site at all)
```

### Key Data Flows

1. **Content to HTML (build time):** Markdown blog posts and Astro page files are compiled into static HTML during `astro build`. This is the primary data flow. All content decisions happen at build time.
2. **Form submission (runtime):** The only user-generated data flow. The site's HTML form sends a POST request directly to a third-party form service. The Oriflect site never processes or stores form data.
3. **Calendly booking (runtime):** Entirely handled by Calendly's embedded widget. The site only provides the container element. All scheduling, confirmation, and calendar sync happen on Calendly's servers.

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 0-10k visitors/month | Static files on any CDN. Zero scaling concerns. Free tier hosting is sufficient. |
| 10k-100k visitors/month | Same architecture. Static sites scale linearly with zero effort. May want image CDN (already handled by Astro image optimization). |
| 100k+ visitors/month | Same architecture still works. Consider adding edge caching headers. Blog search could use a client-side library (Pagefind). This traffic level is unlikely for a consulting site. |

### Scaling Priorities

1. **First bottleneck: Content velocity, not traffic.** The real scaling challenge is creating blog content and updating services, not handling load. The architecture should prioritize easy content authoring (Markdown files, clear frontmatter schema).
2. **Second bottleneck: Form spam.** As the site gains visibility, contact form spam increases. Use honeypot fields from day one and upgrade to CAPTCHA if needed. Third-party form services handle this.

## Anti-Patterns

### Anti-Pattern 1: Reaching for React/Next.js

**What people do:** Use Next.js or a heavy SPA framework for what is fundamentally a brochure site.
**Why it's wrong:** Ships hundreds of KB of JavaScript for a site that has two interactive elements. Slower initial load, worse SEO performance, more complex build/deploy, and unnecessary hydration overhead.
**Do this instead:** Use Astro with zero-JS defaults. Add islands only for the Calendly widget and contact form. Every other element is static HTML.

### Anti-Pattern 2: Building a Custom Form Backend

**What people do:** Set up a server, database, or serverless function to handle contact form submissions.
**Why it's wrong:** Adds infrastructure to maintain, security surface area (input validation, SQL injection, email sending), and deployment complexity for something that receives maybe 5-20 submissions per week.
**Do this instead:** Use a third-party form service (Formspree, Web3Forms, or Netlify Forms). These handle validation, spam filtering, email notifications, and submission storage. Zero backend code.

### Anti-Pattern 3: Overengineering the Blog

**What people do:** Set up a headless CMS, build complex category/tag taxonomies, add search, pagination, related posts, and reading time estimates before publishing a single article.
**Why it's wrong:** Delays launch. The blog section needs to exist with basic functionality. Advanced features can be added when there are actually enough posts to justify them.
**Do this instead:** Start with Astro Content Collections, a simple blog listing page sorted by date, and individual post pages. Add tags/categories only when there are 10+ posts.

### Anti-Pattern 4: Calendly Iframe Instead of JS Embed

**What people do:** Use a raw `<iframe>` to embed Calendly because it seems simpler.
**Why it's wrong:** Loses auto-resize behavior (fixed height causes scrollbars or whitespace), cannot prefill visitor info, cannot track booking events, and cannot style to match the site.
**Do this instead:** Use Calendly's official JavaScript embed snippet. It supports inline, pop-up text, and pop-up widget modes with auto-resize and event callbacks.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| Calendly | JS embed snippet loaded client-side. Inline embed on Contact page, optional pop-up widget as floating CTA on all pages. | Requires Calendly account. Free tier supports one event type. Use `data-url` attribute to point to specific booking page. Test on mobile -- inline embed can be tall. |
| Formspree / Web3Forms | HTML form `action` attribute points to service endpoint. Standard POST with form fields. | Free tier: 50 submissions/month (Formspree) or 250/month (Web3Forms). Add honeypot field for spam protection. Configure email notifications in service dashboard. |
| Google Analytics / Plausible | Script tag in `<head>` via BaseLayout. | Optional. Plausible is privacy-friendly and lightweight (< 1KB). GA requires cookie consent banner in EU. Decide based on privacy stance. |
| @astrojs/sitemap | Astro integration that auto-generates sitemap.xml at build time. | Add to `astro.config.mjs`. Set `site` config. Zero ongoing maintenance. |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| Pages <-> Layouts | Astro slot composition (`<slot />`) | Each page declares its layout in frontmatter. Layout provides header, footer, meta. Page provides content via default slot. |
| Pages <-> Components | Props and slots | Components receive data via Astro props. No global state. No context providers. Simple and explicit. |
| Blog Posts <-> Content Collections | Frontmatter schema validation | Content Collections enforce that every blog post has required fields (title, date, description). Build fails if schema is violated. |
| Contact Form <-> Form Service | HTTP POST (browser-native form submission) | No JavaScript required for basic submission. JS only needed for enhanced UX (inline success message instead of redirect). |
| Any Page <-> Calendly | Client-side JS widget injection | Calendly script loaded asynchronously. Does not block page render. Renders into a designated container div. |

## Build Order (Suggested Implementation Sequence)

The dependencies between components suggest this build order:

1. **Project scaffold + BaseLayout** -- Everything depends on the layout shell (header, footer, nav, meta tags, global styles). Build this first.
2. **Home page** -- Primary landing page. Exercises the layout, section components, and CTA patterns that all other pages reuse.
3. **Services + About pages** -- Static content pages. Reuse section components from home page. Low complexity.
4. **Contact page + Form integration** -- Introduces the first external integration (form service). Includes Calendly embed. Both are isolated islands.
5. **Blog infrastructure** -- Content Collections schema, blog listing page, individual post page template, sitemap generation.
6. **Blog content** -- Write and publish initial posts. This is content work, not engineering.
7. **Polish** -- SEO meta tags, Open Graph images, performance audit, analytics integration.

**Rationale:** Layout first because everything renders inside it. Home page second because it defines the visual vocabulary (hero, CTA, service cards) reused across the site. Contact page before blog because it is the primary conversion mechanism. Blog last because it is the most independent component -- it has no dependencies on other pages and no other pages depend on it.
